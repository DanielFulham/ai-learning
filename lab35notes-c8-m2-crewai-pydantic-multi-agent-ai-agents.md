## Summary

Same framing lab as L34 with a load-bearing addition — `output_pydantic` on structured tasks, `context=[]` for typed inter-task propagation, YAML+CrewBase for a fifth agent alongside four Python-defined ones. Where L34 established the framework baseline, L35 tests whether type-safety-everywhere holds through nested Pydantic across the agent boundary. It does.

Modernised against notebook's shape: watsonx-hosted Granite (skills-network endpoint, non-portable) swapped for Anthropic Claude Haiku 4.5 via CrewAI's native provider; `langchain`/`langchain-community`/`langchain-openai`/`duckduckgo-search`/`databricks-sdk` dropped entirely; `litellm.ssl_verify = False` dropped as a security anti-pattern (MITM vulnerability); watsonx env vars removed; `.env` + `load_dotenv()` at module scope with fail-loud checks on `ANTHROPIC_API_KEY` and `SERPER_API_KEY`; `CREWAI_TRACING_ENABLED=false` in `.env.example`. Every Pydantic field either constrained by a StrEnum picklist (`StoreSection`, `DifficultyLevel`), a validated value object (`PriceEstimate` with `ge=0` and `@model_validator` for `max >= min`), or a typed list. `MealPlanningInputs` TypedDict + `Unpack` at the kickoff-inputs seam via a `build_meal_planning_inputs` factory returning `dict[str, str | int | list[str]]` — no `cast`, no `Any` in the file except at four bounded framework-boundary lines. `isinstance(result, CrewOutput)` narrow at every kickoff site (L34 discipline). Four output artefacts under `outputs/`: `meals.json`, `shopping_list.json`, `shopping_guide.md`, `final_guide.md`.

`LLM(model="anthropic/claude-haiku-4-5", ...)` on CrewAI 1.15.2 auto-routes to the native `AnthropicCompletion` provider when the `anthropic` SDK is present — same routing as L34, but with a corrected declaration: **CrewAI 1.15.2 does NOT install the `anthropic` SDK transitively.** Either `crewai[anthropic]==1.15.2` or explicit `anthropic==0.116.0` in requirements is required to unlock the native path. L34's requirements silently under-declared this; the SDK was present from earlier lab installs. Corrected in L35 and owed as a follow-up commit for L34 (see corrections section below).

Load-bearing finding: **`output_pydantic` succeeds at 4-level nesting on Claude Haiku 4.5 native provider first attempt across all five kickoffs.** `GroceryShoppingPlan` → `ShoppingCategory` → `GroceryItem` → `PriceEstimate` composed with parallel `MealPlan` → `DifficultyLevel` branch, two StrEnums, `list[GroceryItem]` with up to 10 items, `list[ShoppingCategory]` with 3 sections. `.pydantic` populated on every task where declared. Consistent with LLM strength on JSON structure at Haiku 4.5's capability — not evidence the mechanism holds at 5+ levels or with discriminated unions. `output_pydantic` mechanism confirmed empirically as prompt-embedded schema instructions plus post-hoc extraction (per CrewAI docs and observed behaviour) — structurally different from LangGraph's `with_structured_output` which binds schema into the model's tool-calling API. LangGraph retries at model boundary on failure; CrewAI's version fails at framework parser layer with no retry mechanism. The failure mode ("`task.output.pydantic is None` on a `successful` run because the LLM produced text that failed post-hoc validation") is structurally possible but wasn't triggered in these runs.

Second load-bearing finding: **explicit `context=[task]` propagates the upstream `output_pydantic` object as a typed Python object to the downstream task's prompt, not paraphrased text.** Confirmed by verbose stdout: shopping_organizer received `MealPlan(meal_name='Easy Chicken Stir Fry', difficulty_level=<DifficultyLevel.EASY: 'Easy'>, servings=4, ...)` as a real object with typed field access. This is the CrewAI-specific version of what LangGraph gets via TypedDict state — type safety compounds across the pipeline; every layer of Pydantic discipline propagates as typed data, not narrative. The Pydantic models aren't just internal Python validation, they're the wire format between agents.

Sequential-process threading correction (a real mistake in build reasoning worth surfacing): initial claim during construction that implicit sequential threads *all* prior tasks' outputs was **wrong**. CrewAI docs are explicit — *"the output of one task is automatically relayed into the next one."* Sequential auto-relays the immediately previous task only. GitHub issue #2183 requests a `context=False` flag because there's no clean way to suppress the auto-relay from the previous task, which confirms explicit `context=[]` is **additive** on top of the always-inherited previous-task output, not filtering. Practical implication on the five-agent pipeline: `summary_task.context=[meal_planning_task, shopping_task, budget_task, yaml_leftover_task]` is doing real work; without it, only `yaml_leftover_task`'s output would reach summary_agent. The other three would be lost to sequential's one-task-back window.

Third load-bearing finding: **prompt caching is definitively zero across five runs, three pipeline shapes, 45 requests at the largest scale.** Closes L34's deferred cache-runtime-verification item as a real gap rather than measurement noise. CrewAI 1.15.2's native Anthropic provider is not producing `cache_control` markers Anthropic's API registers as cache-eligible, despite `mark_cache_breakpoint()` being wired at source per L34's investigation. Both `cache_creation_tokens` and `cached_prompt_tokens` at zero on every kickoff regardless of prompt size or pipeline depth. During V1 build I overreached on this ("wired but not firing" after two runs) then walked back to "banked open question" after honest pushback, then graduated to definitive gap only after five data points — good discipline shape worth preserving for future findings. Root-cause investigation (turn on `ANTHROPIC_LOG=debug`, inspect outgoing request bodies for `cache_control` blocks) banked separately, closes at source-code level.

L34 established `expected_output` as the strongest lever on completion tokens. L35 sharpens: **`max_iter` is the strongest lever on prompt tokens for search-heavy retrieval agents.** Meal_planner at `max_iter=3` vs `max_iter=1`, same query and inputs, ~80% prompt token reduction (36,428 → 7,369), ~48% completion reduction (819 → 430), same structured output quality. The three-iteration run made 8 Serper searches producing progressively narrower queries; iterations 2-3 added noise, not signal. Bank as a nuance: **only holds for retrieval-and-summarise agents; synthesis agents (no tools, output_pydantic validation-retry semantics) want `max_iter=3` for structured-output retry room.** Same integer, opposite reasoning — per-agent tuning, not global. Post-L35 cost-lever hierarchy: `max_tokens` (defensive backstop, truncation only), `expected_output` (biases completion length but doesn't enforce — L35 observed 400-word target overshot to 700+ on budget_task), `max_iter` (primary lever on tool-use expansion for retrieval agents), `allow_delegation=False` (kills delegation-expansion cost from L34).

Fourth load-bearing finding: **cross-agent arithmetic drift is a real quality issue that a well-instructed summary_agent surfaces rather than hides.** Shopping_organizer's `total_budget` mirrored user input ($20-$25) rather than reflecting its own section sums ($19.75-$29.00). Budget_advisor's Serper-grounded analysis computed $14.95-$21.90 total — meaningfully different from both. Summary_agent, given `expected_output` prose containing "Preserve the shape of each upstream section; do not re-analyze or re-price. This is a compilation, not new analysis," handled the discrepancy honestly: reported both upstream numbers ("Total Estimated Cost: $19.75 - $29.00" and "Total Estimate: $14.95 - $21.90 (per budget analysis)") and reconciled with "Both estimates fit within the $25 budget on the lower end." Didn't pick a winner, didn't invent a third number, didn't hide the discrepancy. Bank as a production-shape pattern: **downstream synthesis agents can *reconcile* conflicts between upstream agents when instructed to compile, not re-derive.** Same principle as showing users a confidence interval rather than a single point estimate — honesty about upstream uncertainty is a feature.

Fifth finding worth surfacing separately: **prompt instructions asking an agent to *not* use an available capability are unreliable.** Budget_task's `expected_output` said "Assume upstream shopping list is authoritative; do not re-price items unless suggesting a substitution." Budget_advisor ignored this, ran three Serper searches, produced sharper Serper-grounded prices. The LLM chose to be more useful than the constraint asked. Reliable mechanism is capability removal (drop the tool from the Agent constructor), not prompt discipline. Same category as `allow_delegation=False` being reliable vs "please don't delegate" in the prompt. This refutes a V2 architectural suggestion I made during build ("remove Serper from budget_advisor because it compounds hallucination") — the tool's presence is the point, and the resulting cross-agent drift is what the summary_agent's compile discipline honestly surfaces.

Silent Pydantic extra-field acceptance on Agent (L34 finding) generalises to CrewBase's `Task(config=task_config)` and `Agent(config=agent_config)` — a typo in the YAML propagates silently through the same `extra="allow"` Pydantic surface. V1 avoids the hazard on the YAML side by pulling fields explicitly from the loaded dict with `.get(..., default)` for optionals: `Agent(role=agent_config["role"], goal=agent_config["goal"], backstory=agent_config["backstory"], llm=self._llm, max_iter=agent_config.get("max_iter", 3), allow_delegation=agent_config.get("allow_delegation", False), verbose=agent_config.get("verbose", False))`. Trade-off: YAML becomes "prompt content plus known-field config," not "arbitrary Agent kwargs." Same production discipline as YAML-vs-code separation of concerns — YAML for what non-Python people edit, Python for schema enforcement.

CrewBase framework friction is real and worth naming honestly: **`@CrewBase`, `@agent`, `@task` are officially incompatible with strict pyright.** GitHub issue crewAIInc/crewAI#2539 (April 2025, still open) titled "[FEATURE] No Errors in Pyright for Example Code" documents the exact errors V1 hit. CrewAI's own docs at docs.crewai.com/en/concepts/agents include `# type: ignore[index]` markers in scaffolded example code. That's the framework-canonical pattern — not a workaround someone in the community invented, the official recommendation. V1 lands four bounded `# type: ignore` markers with specific error codes at the CrewBase boundary — three at `LeftoversCrew` usage sites in `grocery_planner.py`, one at `self.leftover_manager()` inside `leftover.py`. Rest of the file is `type: ignore`-free including the explicit-kwarg YAML extraction. Bank as an L35 finding: **CrewBase's decorator machinery is incompatible with strict-pyright discipline in the current release, and the framework's own docs use `# type: ignore` markers.** V2 onion port will scope all four markers to `infra/crewai_orchestrator.py`, keeping application-layer code pyright-clean.

Small design instinct that matters: **YAML config in agent frameworks is a review-and-rollback improvement, not a runtime editability mechanism.** During build I answered a question about YAML enabling no-redeploy prompt updates — the honest answer is that CrewBase loads YAML at class instantiation via `yaml.safe_load()`, once per process. Long-running services (Lambda warm containers, FastAPI backends) hold the loaded configs for the process lifetime; edits to YAML on disk take effect only at process restart. What YAML *does* unlock is smaller review surface for prompt changes, safer rollback, and config-as-data separation from code-as-behaviour — which are real production wins in review time and deploy risk, distinct from runtime editability. Genuine no-redeploy prompt updates require a remote config store (Redis/DynamoDB/feature-flag service) polled or subscribed to at runtime. Bank as an AI_PATTERNS.md-worthy finding for production interviews and Hooperman/Observability future work.

## Post-Course-8 retrospective (V1 authoring)

I built V1 with type-safety-first instincts carried from .NET and the L27-L34 sequence. Every boundary that could be structured with Pydantic got structured — StrEnums for closed picklists, `PriceEstimate` value object with a `@model_validator` for the `max >= min` invariant, `TypedDict` + `Unpack` at the kickoff-inputs seam, `isinstance` narrows at every framework return-type union. The gamble was that CrewAI's `output_pydantic` mechanism would round-trip through Claude Haiku 4.5 cleanly at 4-level nesting; that gamble paid off empirically. The related gamble was that `context=[]` would propagate typed pydantic objects, not paraphrased text; that also paid off. Together these are the load-bearing structural finding of L35 — **type safety compounds across the CrewAI pipeline**, and the discipline built into V1 gives you production-shape agent-to-agent contracts, not just internal Python validation.

The V1 build cost more than I expected in framework-boundary friction. `@CrewBase` decorator opacity produced hours of pyright argument that resolved to "the framework's own docs use `# type: ignore` markers, this is a known open issue" (crewAIInc/crewAI#2539). During the build I made two wrong turns worth naming: an early confident claim that implicit sequential threading passes *all* prior tasks' outputs (turned out to be only the immediately previous task), and an early confident claim that CrewAI's prompt caching was "wired but not firing" after two runs (walked back to "banked open question" after honest pushback, graduated to definitive gap only after five data points across three pipeline shapes). Both corrections came from pushing back on my own reasoning after evidence contradicted it. Bank the discipline for future findings: **directional evidence isn't a finding; a single measurement isn't a finding; a claim survives only after multiple runs of different shapes cluster around the same result.** L34's median-of-N applied across shapes, not just repetitions.

The framework-vs-type-safety tension is real production-shape material. CrewAI is designed for scaffolded `crewai create crew` projects where the CLI templates the whole thing and pyright compliance isn't the discipline being applied. Using CrewAI in a hand-written type-safe codebase is a fight the framework didn't design for. V1 lands the fight by confining `# type: ignore` markers to four lines at the CrewBase boundary, keeping the application-layer code type-honest. V2's onion port will scope those four to `infra/crewai_orchestrator.py`, matching L34-1's V2 shape and the broader pattern set. This is exactly what I'd write up for an interview answer on "how do you use a framework whose type story is worse than yours."

The exercises got deferred to preserve cert velocity and to prep V2 test material. Exercise 1 (nutrition_analyst) is a new problem domain — nutrition analytics without a grounding data source (USDA food composition tables, macros per ingredient) would layer nutritional-value drift on top of price drift, compounding the cross-agent arithmetic drift finding rather than producing new architectural insight. Better built post-cert against Supporter 360 or a similar real product context. Exercise 2 (WeeklyMealPlan et al) is the cross-day type-safety stress test — V1's single-meal Pydantic composition is proven at 4 levels; weekly composition with bulk-buying invariants across the day boundary is the natural next stress. Deferred to V2's onion port because that's where application-layer validators can honestly enforce the cross-day invariants without polluting domain models with framework-specific concerns.

CrewAI vs LangGraph as data-flow shapes is the most useful cross-cutting observation from the build. CrewAI takes a **push** model — each task's output pushes forward automatically (only the immediately previous, per corrected finding), and explicit-listed tasks add on top. Simple mental model for two-task pipelines, but power over what a task sees is limited. LangGraph takes a **pull** model — nodes explicitly read state keys they name, no auto-inheritance. LangGraph's model matches the type-safety instinct better (explicit-first by construction, TypedDict state as the wire format between nodes); CrewAI's model optimises for framework ergonomics on short pipelines at the cost of explicit-per-task filtering for anything longer. Neither is inherently better; they encode different priors. Bank as blog-worthy AI_PATTERNS.md content — data-flow-between-agents as a first-class architectural axis, with CrewAI push vs LangGraph pull as the two canonical shapes.

## Key Takeaways

- **`output_pydantic` mechanism confirmed** as prompt-embedded schema + post-hoc extraction. Structurally different from LangGraph's native tool-calling; failure mode is "pydantic returns None on 'successful' run." Worked cleanly at 4-level nesting on Haiku 4.5 across all five kickoffs. Not evidence it holds at 5+ levels or with discriminated unions.

- **Type safety compounds across the CrewAI pipeline.** `output_pydantic` on task N + `context=[task_N]` on task N+1 propagates the typed pydantic object, not stringified prose. The Pydantic models are the wire format between agents.

- **Sequential threading passes only the immediately previous task's output.** `context=[]` is additive on top of that, not filtering. GitHub issue crewAIInc/crewAI#2183 (Feb 2025) is a feature request for a `context=False` flag because there's no current way to suppress the auto-relay.

- **Prompt caching definitively zero across five runs at three pipeline scales.** Closes L34's deferred cache-runtime-verification as a real gap in CrewAI 1.15.2's native Anthropic provider, not measurement noise. Root cause requires SDK-level source inspection.

- **`max_iter` is the primary lever on prompt tokens for retrieval agents.** Meal_planner `max_iter=3` → `max_iter=1` produced ~80% prompt reduction, same output quality. Nuance: only for retrieval-and-summarise; synthesis agents want `max_iter=3` for validation-retry room. Post-L35 cost-lever hierarchy: max_tokens, expected_output (biases not enforces), max_iter, allow_delegation.

- **Cross-agent arithmetic drift is real; a well-instructed summary_agent reconciles honestly.** Shopping_organizer $19.75-$29 vs budget_advisor $14.95-$21.90 → summary_agent surfaced both, didn't invent a third. "Compile don't re-analyze" permits reconciliation of upstream conflicts.

- **Prompt instructions asking an agent NOT to use an available capability are unreliable.** Reliable mechanism is capability removal (drop the tool), not prompt discipline. Same class as `allow_delegation=False` being reliable vs "please don't delegate."

- **CrewBase framework friction is a known open issue** (crewAIInc/crewAI#2539). Framework's own docs use `# type: ignore` markers. V1 confines four bounded ignores to the CrewBase boundary; V2 onion port scopes them to `infra/crewai_orchestrator.py`.

- **YAML config unlocks review-and-rollback improvements, not runtime editability.** Genuine no-redeploy prompt updates require a remote config store polled at runtime; YAML in the CrewBase pattern is one step short of that.

- **L34 requirements correction owed:** `crewai==1.15.2` does NOT install `anthropic` transitively. Pin `anthropic==0.116.0` explicitly. L34's technical findings unaffected.

- **CrewAI vs LangGraph as data-flow shapes.** Push vs pull — CrewAI auto-relays previous-task output, explicit context is additive; LangGraph nodes read state keys explicitly, no auto-inheritance. LangGraph's shape matches type-safety instincts more naturally.
